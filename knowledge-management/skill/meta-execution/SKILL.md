> **核心理念**：把事情一次做对，而不是反复返工。

## 技能定位

这是一个**元技能（Meta-Skill）**——它不是用来做某件具体事情的，而是用来确保**任何事情都能做好**的工作流程。

每次接到任务时，这个技能应该**自动激活**，贯穿整个任务生命周期。

---

## 🚨 Critical Rules（不能绕过的禁止项）

- ❌ **禁止在续接会话时跳过上下文恢复** — 摘要只有结论没有流程，必须恢复完整上下文
- ❌ **禁止跳过启动准备的经验检索** — 回顾相关历史经验是强制步骤，不是可选步骤
- ❌ **禁止第2次失败时重复相同操作** — 3-Strike 规则：第2次失败必须换方法，不是重试
- ❌ **禁止在未过交付审查三阶段前交付** — 规格符合 → 质量审查 → 输出规范，缺一不可
- ❌ **禁止在遇到阻碍时直接放弃** — 中途中断前必须记录当前状态和待处理问题

## 动态激活机制

### 评分标准（C1-C7）

| 编号 | 标准 | 识别信号 |
|:-----|:-----|:---------|
| C1 | 多阶段工作流（3+步骤） | 任务包含多个依赖步骤 |
| C2 | 外部可见交付物 | 产出物会被他人看到 |
| C3 | 涉及部署/发布 | 修改会影响生产环境 |
| C4 | 无人值守执行 | 用户不在场，无法实时纠偏 |
| C5 | 数据敏感性要求 | 涉及敏感信息处理 |
| C6 | 用户强调质量 | 用户话语中隐含质量期望 |
| C7 | 高复杂度任务 | 预计步骤>5步 |

### 触发规则

```
命中 ≥ 2 项 → 完整激活（Phase 0-4 全流程）
命中 1 项   → 轻量模式（仅 Phase 3 交付审查）
命中 0 项   → 不激活（简单任务直接执行）
```

## Phase 0: 续接恢复（Resume）

> **适用场景**：新会话是从之前会话延续的

### 核心洞察

> **会话摘要 ≠ 完整上下文**。摘要给结论，但缺少流程、约束、已踩的坑。不恢复完整上下文就执行 = 蒙眼开车。

### 检测信号（命中任一条 = 续接会话）

| 信号 | 识别方式 |
|:-----|:---------|
| 上下文溢出恢复 | 系统提示这是从之前会话延续的 |
| 会话摘要存在 | 初始上下文中包含之前对话的总结/分析 |
| 用户续接意图 | 用户说"继续"、"接着做"、"上次没做完" |
| 任务状态文件存在 | 存在之前会话的进度文件/检查点 |
| 相关历史线程 | 近期有同主题的对话记录 |

### 强制动作

```
Step 1: 检测续接信号 → 任一命中 → 进入 Phase 0 流程
Step 2: 识别任务领域 → 从摘要中识别涉及的任务类型/技能
Step 3: 恢复完整上下文 → 加载相关技能文档、读取进度文件
Step 4: 状态声明 → 向用户简述：已完成什么、当前状态、下一步
Step 5: 确认后继续 → 用户确认或给出"继续"指令 → 进入 Phase 1
```

### 上下文恢复清单

```
□ 原始任务目标是什么？
□ 已经完成了哪些步骤？
□ 做过哪些关键决策？为什么？
□ 踩过哪些坑？有什么约束？
□ 下一步应该做什么？
□ 用户有什么特殊偏好/要求？
```

### 反模式

| 不要做 | 应该做 |
|:-------|:-------|
| 看到摘要写"Step 1已完成"就直接从Step 2开始 | 恢复完整流程，确认还有哪些步骤 |
| 用户说"继续"就立刻动手 | 先检测续接信号，恢复上下文 |
| "上个会话做过了，我记得" | 每个新会话都重新加载文档，记忆不可靠 |
| 信任摘要中的"已完成"状态 | 独立验证产出，状态可能是自报告的 |

## Phase 1: 启动准备（Prepare）

> **适用场景**：开始执行任务之前

### 目标

确保理解正确、方向正确，避免"做完才发现理解错了"。

### 🔴 必做核心（每次任务都要过）

```
□ 需求理解：我真的理解用户想要什么吗？有模糊点吗？
□ 技能检查：有匹配的专业技能可以辅助吗？
□ 经验回顾：从记忆系统中检索与当前任务相关的历史经验
□ 方案预演：脑中走一遍流程，有没有明显的问题？
```

### 经验回顾执行方式

使用系统提供的记忆检索能力，搜索与当前任务相关的历史经验。

> 具体工具和参数取决于当前系统的记忆能力实现。

### 🟡 复杂任务扩展（按需启用）

| 检查项 | 触发条件 | 说明 |
|:-------|:---------|:-----|
| 苏格拉底式提问 | 需求模糊 | 先问清范围/预期/约束，再动手 |
| 设计先行 | 大任务 | 分块展示方案，确认后再执行 |
| 任务原子化 | 复杂任务 | 拆分为可独立验证的小任务 |
| 上下文预算评估 | 长任务 | 判断是否需要分阶段检查点 |
| 三文件规划 | >5次操作 | 创建 task_plan.md / findings.md / progress.md |

### 上下文消耗评估

| 信号 | 轻量 | 中等 | 重度 |
|:-----|:-----|:-----|:-----|
| 待办项数 | ≤5 | 6-10 | >10 |
| 涉及技能数 | 0-1 | 2-3 | >3 |
| 独立子阶段 | 1-2 | 3-4 | ≥5 |

**判定**：重度任一条 → 启用阶段检查点 | 中等≥2条 → 考虑检查点 | 其他 → 正常执行

详细协议见 `references/context-budget-protocol.md`

### 红旗警示（这些想法 = 合理化逃避）

| 危险想法 | 正确做法 |
|:---------|:---------|
| "这只是个简单问题" | 花5秒扫一遍检查清单 |
| "我记得怎么做" | 查阅文档确认，记忆不可靠 |
| "这个流程太重了" | 至少过 Phase 3，简单任务也需要交付审查 |

## Phase 2: 执行过程（Execute）

> **适用场景**：任务执行过程中

### 目标

边做边检查，而不是做完再检查。

### 核心行为

1. **边界自觉**：这个问题超出我的能力了吗？需要求助吗？
2. **分步验证**：复杂任务分步执行，每步检查
3. **及时沟通**：发现预期外的情况，及时告知用户
4. **2-Action Rule**：每执行2次查看/搜索后，立即保存关键发现
5. **WAL Protocol**：重要信息先持久化，再继续（内存是易失的）

### Read vs Write 决策矩阵

| 情况 | 动作 |
|:-----|:-----|
| 刚写完内容 | 不要重读（内容还在上下文中） |
| 看了外部信息 | 立刻记录关键发现 |
| 开始新阶段 | 读取计划/已有发现 |
| 发生错误 | 读相关文档排查 |

### 3-Strike Error Protocol

```
第1次失败: 诊断 & 修复 → 识别根因 → 定向修复
第2次失败: 换方法 → 不同工具/路径/方案（⛔ 不重复相同失败操作）
第3次失败: 资源组合 → 质疑假设 + 搜索方案 + 组合方法
第4-9次: Relentless Mode → 尝试5-10种不同方案
第10次后: 升级给用户 → 列出尝试过的方案 + 具体错误
```

> **核心规则**：`if action_failed: next_action != same_action`

## Phase 3: 交付审查（Review）

> **适用场景**：把结果交给用户之前

### 目标

拦截所有能发现的问题，不让用户帮我做质检。

### 三阶段审查

```
┌──────────────────┐      ┌──────────────────┐      ┌──────────────────┐
│  Stage 1: Spec   │  →   │  Stage 2: Quality │  →   │  Stage 3: Output │
│  规格符合        │      │  质量审查         │      │  输出规范        │
└──────────────────┘      └──────────────────┘      └──────────────────┘
```

**Stage 1: 规格符合审查**

| 检查项 | 问题 |
|:-------|:-----|
| 完整性 | 用户要求的每一条都有对应输出吗？ |
| 无遗漏 | 有没有漏掉"顺便XXX"的附加要求？ |
| 无越界 | 有没有做用户没要求的事？ |

**Stage 2: 质量审查**

| 检查项 | 问题 |
|:-------|:-----|
| 正确性 | 有没有明显的事实/逻辑错误？ |
| 清晰性 | 用户能看懂吗？ |
| 美观性 | 格式整齐吗？ |

**Stage 3: 输出规范检查**

| 输出类型 | 检查内容 |
|:---------|:---------|
| 网页/HTML | 需要打开预览吗？ |
| 图片 | 需要展示给用户吗？ |
| 文件输出 | 放在合适的位置了吗？ |
| 代码 | 能直接运行吗？有遗漏的依赖吗？ |
| 信息/分析 | 不确定的内容标注了吗？ |
| 任何交付物 | 站在用户角度审视过了吗？ |

> **核心原则**: 宁可多花10秒检查，不要让用户帮我做质检。

## Phase 4: 事后复盘（Reflect）

> **适用场景**：任务完成之后

### 目标

从每次执行中学习，持续进化。

### 快速自检（每次任务后）

```
□ 探索了新的路径或发现了新信息？
□ 遇到了不明显的约束或规律？
□ 踩了坑（某种方式行不通）？
□ 发现了某类需求的通用解法？
□ 被用户纠正或任务返工了？
```

### 触发规则

| 自检结果 | 动作 |
|:---------|:-----|
| ≥2条为是 | 激活系统复盘能力，进行深度复盘 |
| 1条为是 | 快速记录关键发现 |
| 全否 | 静默，无需额外动作 |

### 激活系统复盘能力

当需要深度复盘时，按以下优先级查找系统中可用的复盘机制（不阻塞用户）：

| 优先级 | 机制类型 | 查找方式 | 激活方式 |
|:-------|:---------|:---------|:---------|
| 1 | **Skill** | 查找名称含 `reflect`、`learn`、`review`、`retrospective`、`memory-trigger`、`skill-trigger` 的技能 | `use_skill(skill_name)` |
| 2 | **Subagent** | 查找专门的复盘/学习代理 | `use_subagent(agent_name, task)` |
| 3 | **Tool** | 查找 `save_memory`、`log_insight`、`record_lesson` 等工具 | 直接调用工具 |
| 4 | **文件系统** | 写入指定的经验/教训目录 | 写入 `lessons/`、`insights/` 等目录 |

> 均为**后台异步执行**，触发后不阻塞用户，由后台 Agent 自动判断是否创建/更新/删除记忆或技能。

**触发优先级**：
- 有明显失败/错误 → 优先触发记忆类复盘机制
- 完成高复杂度任务 → 优先触发技能类复盘机制
- 两者均满足 → 可同时触发

**如果系统无任何复盘机制**：直接在当前响应中输出复盘总结：

```markdown
## 📝 任务复盘

### 做得好的
- ...

### 踩的坑
- ...

### 下次改进
- ...
```

### 沉淀分级规则

| 问题类型 | 沉淀方式 |
|:---------|:---------|
| 关键问题（导致可见缺陷/同类2次+） | ⚠️ 强制规则（用"必须/禁止"） |
| 一般问题（影响效率不影响正确性） | 建议/经验（用"推荐/建议"） |

## 通用踩坑模式

以下是常见的执行陷阱，详见 `references/common-pitfalls.md`：

| 模式 | 说明 | 防护 |
|:-----|:-----|:-----|
| 静默失败 | 错误被忽略，继续执行 | 每步检查返回值/状态 |
| 假完成 | 声称完成但实际未执行 | 验证实际产出而非状态 |
| 清单即全集 | 认为现有清单是完整的 | 批量操作前全量扫描 |
| 声明即执行 | 写了规则≠会遵守规则 | 规则需要有执行步骤 |
| 元工具盲区 | 管理工具无法管理自己 | 修改元规则后外部审计 |
