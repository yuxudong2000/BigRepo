# 上下文预算管理协议

> 上下文是不可再生资源——用完就没了。必须像管理预算一样管理它。

## 为什么需要预算管理

长任务中，上下文窗口逐渐填满会导致系统性质量衰减：

- **注意力稀释**：每个信息分到的关注度降低，早期指令被"遗忘"
- **中间遗失**：对开头和末尾关注高，中间关注低
- **隐式压力**：剩余空间越少，越倾向省略检查步骤

---

## 预算评估（任务开始前）

### 判定表

| 信号 | 轻量 | 中等 | 重度 |
|:-----|:-----|:-----|:-----|
| 待办项数 | ≤5 | 6-10 | >10 |
| 涉及技能/领域数 | 0-1 | 2-3 | >3 |
| 独立子阶段 | 1-2 | 3-4 | ≥5 |

### 判定规则

```
命中"重度"任一条 → 启用阶段检查点 + 考虑分会话
命中"中等" ≥ 2条 → 启用阶段检查点
其他              → 正常执行，无需特殊管理
```

### 判定示例

| 任务 | 待办数 | 领域数 | 子阶段 | 判定 | 策略 |
|:-----|:-------|:-------|:-------|:-----|:-----|
| "帮我写个脚本" | 3 | 0 | 1 | 轻量 | 正常执行 |
| "优化网站+部署" | 7 | 2 | 3 | 中等(2条) | 阶段检查点 |
| "深度调研+写报告+部署" | 10 | 4 | 4 | 重度 | 检查点+分会话 |

---

## 阶段检查点（会话内）

### 触发条件

预算评估为"中等"或"重度"时启用。

### 检查点格式

每个阶段完成后，在响应末尾附加：

```
───────── 阶段 N 检查点 ─────────
已完成：
  - [阶段 1] 任务描述 → 结果摘要（一句话）
  - [阶段 2] 任务描述 → 结果摘要（一句话）

当前状态：
  - 关键产出物：file_a.py(已改), file_b.md(新建)
  - 关键决策：选择了方案X，因为...

下一步：
  - [阶段 N+1] 具体任务描述
  - 关键约束：不能修改xxx / 必须保持yyy

风险备忘：
  - 已踩的坑：zzz（避免重复）
──────────────────────────────────
```

### 检查点约束

| 约束 | 要求 | 原因 |
|:-----|:-----|:-----|
| 长度 | 200-400字 | 过长则适得其反 |
| 压缩比 | >10:1 | 前面大量执行细节压缩为简短摘要 |
| 必含字段 | "下一步" | 确保方向明确 |
| 累积性 | 包含所有已完成阶段的摘要 | 新检查点替代旧的 |

### 为什么有效

1. **注意力锚定**：检查点在最近位置，会被优先关注
2. **冗余裁剪**：已完成细节被压缩，释放注意力
3. **指令刷新**：下一步被重新声明，等效于重新下达指令
4. **认知重置**：创造"新起点"感，而非"漫长任务的尾巴"

## 多会话接力（跨会话）

### 触发条件（任一满足）

1. 预算评估为"重度"，且执行到中段后感知到质量下降风险
2. 执行中出现自我监测信号（见下方）
3. 待完成项还有 >5 个，但已执行很长时间

### 自我监测信号（判断是否正在"虎头蛇尾"）

以下信号出现 ≥2 个时，应考虑分会话：

| 信号 | 表现 |
|:-----|:-----|
| 输出变短 | 同等复杂度的步骤，后期输出明显比前期短 |
| 检查省略 | 开始跳过交付前检查，或检查变敷衍 |
| 步骤合并 | 把本应分步执行的任务合并为一步 |
| 细节丢失 | 忘记之前讨论的约束或决策 |
| 急于结束 | 出现"最后一步"的措辞但实际还有多步 |

### 接力文件格式

```markdown
# 任务接力检查点

生成时间: YYYY-MM-DD HH:MM
阶段进度: N/M

## 原始任务

[完整复制用户的原始需求描述]

## 已完成

- [阶段 1] 任务描述 → 结果摘要
  - 产出: file_path (改动说明)
- [阶段 2] 任务描述 → 结果摘要
  - 产出: file_path (改动说明)

## 待执行

- [阶段 N+1] 具体任务描述
  - 子步骤1: xxx
  - 子步骤2: xxx
- [阶段 N+2] 具体任务描述

## 关键上下文（必须携带）

- 决策: 为什么选择方案X（不能改回方案Y）
- 约束: 不能修改xxx，因为...
- 踩坑: yyy会导致zzz，已验证

## 用户偏好

- 风格/格式要求: xxx
- 质量期望: xxx
```

### 接力流程

```
Step 1: 检测到需要接力
  → 完成当前阶段的收尾工作
  → 生成接力文件

Step 2: 告知用户
  "这个任务比较复杂，为了保证后续阶段的执行质量，
   我已生成检查点文件。建议在新会话中说：
   '继续执行，检查点在 {path}'
   我会从阶段 N+1 开始，以完整的注意力继续。"

Step 3: 新会话启动
  用户：继续执行，检查点在 xxx
  Agent：
    1. 读取检查点文件
    2. 简要复述待执行内容，确认理解正确
    3. 从断点开始执行
```

### 注意事项

- 接力是**建议**而非强制——尊重用户选择
- 接力文件必须自包含——只读这个文件就能完全恢复上下文
- 关键决策和约束最容易在接力中丢失，必须显式列出
