# Product Thinking: 产品思维

> **核心理念**：每个人的所有交付物都是给别人用的，都可以理解成一种"产品"。
> 产品的价值在于满足用户需求。交付物是"顺你的意"，而非"按我的来"。

---

## 技能定位

这是一个**用户主动触发的产品思维分析工具**。用户显式调用后，Agent 会围绕交付物进行结构化分析，并将分析过程和结论清晰输出给用户。

本技能基于《产品思维》系列文章和《研效产品经理修炼指南》提炼而成。

**执行方式**：收到用户请求后，主动输出四问框架的分析结果，给出明确的优化建议。

---

## 触发条件

用户主动发起以下类型请求时触发：

- 「帮我用产品思维 Review 这个设计/方案/页面」
- 「从用户角度分析一下这个交付物」
- 「这个设计有没有反人性的地方？」
- 「帮我做场景分析」
- 「用产品经理视角看看这个」
- 用户提供交付物（网页/文档/PPT/方案/界面截图）并希望获得产品视角反馈

适用于所有类型的交付物：

- 网页、界面、可视化、dashboard
- 文档、报告、分析、总结
- PPT、Keynote、演示文稿
- 图片、图表、设计稿
- 代码、脚本、配置
- 建议、方案、计划
- ...任何需要交付给用户的产物

## 核心原则

### 金句速记

| 原则                                   | 来源                                 |
| -------------------------------------- | ------------------------------------ |
| **"顺你的意"而非"按我的来"**           | 产品经理 vs 销售的区别               |
| **产品不足，营销补；营销不足，渠道补** | 产品是第一位的                       |
| **先有场景，后有产品**                 | 场景决定产品形态                     |
| **用服务视角看产品**                   | "用户要什么服务"而非"我能做什么产品" |

### 销售思维 vs 产品思维

| 维度     | 销售思维 ❌        | 产品思维 ✅          |
| -------- | ------------------ | -------------------- |
| 核心     | "按我的来"         | "顺你的意"           |
| 表现     | 靠说服让用户接受   | 让用户自己体验到价值 |
| 关注点   | 我的实现方案多酷   | 用户用起来有多爽     |
| 典型表现 | 功能堆砌、视觉炫技 | 信息精简、交互最短   |

## 四问框架

**收到用户请求后，围绕交付物依次分析以下4个维度，并将结论输出给用户：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     产品思维四问框架                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   问题4：场景                                                    │
│   ├── 用户什么时候、在哪里使用这个交付物？                         │
│   ├── 用什么方式？（阅读/操作/展示/分享）                         │
│   ├── 花多长时间？（30秒速览/5分钟深入/反复查阅）                 │
│   └── 使用频率？（一次性/每天/每周/偶尔）                         │
│                        ↓                                        │
│   问题3：用户画像                                                │
│   ├── 用户带着什么预期来的？                                     │
│   ├── 他想完成什么任务？                                         │
│   └── 他的技术背景/认知水平如何？                                 │
│                        ↓                                        │
│   问题2：欲望/痛点                                               │
│   ├── 满足感：用户想获得什么"爽"的感觉？                          │
│   │   （如：一眼看到关键信息、炫酷的展示效果）                     │
│   └── 恐惧感：用户想消除什么焦虑/担忧？                           │
│       （如：担心遗漏重要信息、担心操作复杂）                       │
│                        ↓                                        │
│   问题1：交付物设计决策                                          │
│   ├── 基于以上，优先呈现什么？                                   │
│   ├── 主动省略/弱化什么？                                        │
│   ├── 默认呈现形式是什么？                                       │
│   └── 用户获取价值的路径如何最短？                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 输出格式示例

收到用户请求后，输出如下结构：

```markdown
## 🔍 产品思维分析

### 场景判断
[描述用户使用场景、时间、设备、频率]

### 用户画像
[描述目标用户的预期、任务、背景]

### 欲望/痛点
- 满足感：[用户想获得什么]
- 恐惧感：[用户想消除什么]

### 设计决策建议
- ✅ 应该突出：[...]
- ❌ 应该弱化/删除：[...]
- 🔄 交互路径优化：[...]

### 发现的反模式
- [具体问题 + 修复建议]
```

## 服务视角转换

### 语言转换

把 **"我能提供什么产品"** 转换成 **"用户要什么服务 + 我能如何提供"**

| 实现者视角 ❌        | 用户服务视角 ✅                    |
| -------------------- | ---------------------------------- |
| 我做一个能力树可视化 | 用户想快速了解AI助手的能力变化     |
| 我做一个神经网络布局 | 用户想一眼看到关键的新增/变化      |
| 我把三棵树都展示出来 | 用户在30秒内想看到"今天有什么不同" |

### 案例：共享单车的启示

> 用户要的不是自行车，而是"1-5公里范围内低成本快速到达目的地"的服务。

方案演进：

1. 卖自行车给用户 → 传统思路
2. 开租赁店让用户租 → 进化思路
3. 用户需要时送到手上，用完拿走 → **ofo/摩拜的服务思维**

## 场景的重要性

### 场景定义

> **场 = 时间 + 空间**：用户在这个空间里停留和消费
> **景 = 情景 + 互动**：用户必须做的事、必须完成的交互

### 场景决定设计

| 场景           | 设计决策                           |
| -------------- | ---------------------------------- |
| 早上3分钟速览  | 信息密度低、变化高亮、核心数据前置 |
| 工作中深入分析 | 信息完整、层级清晰、支持钻取       |
| 向他人展示成果 | 视觉冲击、故事线清晰、关键成就突出 |
| 手机碎片时间   | 卡片化、可滑动、单手操作友好       |

### 场景的金句

> 如果没想清楚场景，你写的再好，那也只是"**自己想说的**"；
> 如想清楚了，你则会写出"**别人想听的**"。

## 进阶框架（来自《研效产品经理修炼指南》）

### 工程思维 × 产品思维结合

两种思维各有优势，最佳实践是结合使用：

```
领域模型 → 用户服务蓝图 → 用户体验地图/用户旅程地图
```

1. **领域模型 → 用户服务蓝图**：把领域模型翻译成完整的工作流程
2. **用户服务蓝图 → 用户体验地图**：在每个场景下细化用户访问行为，分析痛点

### 用户体验五层次

| 层次   | 内容               | 优先级 |
| ------ | ------------------ | ------ |
| 战略层 | 用户需求、产品目标 | ★★★★★  |
| 范围层 | 产品规格功能       | ★★★★   |
| 结构层 | 产品流程结构       | ★★★    |
| 框架层 | 页面布局           | ★★     |
| 表现层 | 视觉设计           | ★      |

### 简约设计四法

| 方法     | 说明                                 |
| -------- | ------------------------------------ |
| **删除** | 大胆删除非核心功能，只为主流用户设计 |
| **组织** | 分块组织，空间/时间分层              |
| **隐藏** | 不常用但不能少的功能隐藏             |
| **转移** | 向其他设备转移复杂操作               |

### 交互设计五原则

| 原则     | 定义                 |
| -------- | -------------------- |
| **示能** | 一看就知道怎么用     |
| **意符** | 表达意思的符号       |
| **约束** | 设置阻塞防止误操作   |
| **映射** | 把多个功能封装成一个 |
| **反馈** | 每次操作都有及时反馈 |

> 详细框架见 `references/pm-cultivation-guide.md`

## 常见反模式

### ❌ 功能堆砌

**症状**：把所有能做的功能都展示出来
**诊断**：没有问"用户80%的场景需要哪20%的功能"
**修复**：识别核心场景，其他功能隐藏或弱化

### ❌ 视觉炫技

**症状**：优先考虑"这个效果很酷"
**诊断**：从实现者视角出发，而非用户视角
**修复**：问"这个效果帮助用户完成什么任务？"

### ❌ 信息过载

**症状**：一个页面塞太多内容
**诊断**：没有区分信息优先级
**修复**：根据场景确定默认展示什么、折叠什么、隐藏什么

### ❌ 交互路径长

**症状**：用户需要多次点击才能完成任务
**诊断**：没有从"用户任务"出发设计流程
**修复**：画出用户任务流程图，找最短路径

## 分析完成后的输出规范

分析结束后，给出可操作的改进清单：

1. **如果用户只有30秒，他能看到最重要的信息吗？**
2. **用户完成核心任务需要几次点击？能减少吗？**
3. **页面上有没有"我觉得酷但用户用不到"的东西？**
4. **如果让用户选择，他会保留哪些、删掉哪些？**
5. **这个设计解决了用户的什么痛点或满足了什么爽点？**
