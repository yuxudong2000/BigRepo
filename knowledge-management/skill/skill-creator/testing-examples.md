# Skill 测试示例

本文档展示如何使用 TDD 方法测试 Skill。

---

## 测试流程概览

```
RED (基线测试) → GREEN (编写Skill) → REFACTOR (查漏补缺)
      ↓                  ↓                    ↓
   观察失败            让测试通过           找到新漏洞
```

---

## 示例1：创建一个"代码审查"Skill

### Phase 1: RED — 观察失败行为

#### 1.1 设计压力场景

```markdown
**场景描述**：
用户提交了一段代码，要求进行代码审查。

**预期行为**：
- AI 应该系统化地检查代码质量
- 应该检查多个维度（功能/性能/安全/可维护性）
- 应该提供具体的改进建议

**常见错误**：
- 只看一遍代码就给出结论
- 只关注功能，忽略其他维度
- 给出模糊的建议（"代码还可以"）
- 不检查边界情况
```

#### 1.2 运行基线测试

```markdown
**测试代码**：
```python
def process_user_input(user_id, amount):
    user = get_user(user_id)
    user.balance += amount
    save_user(user)
    return "Success"
```

**测试请求**：
"请帮我审查这段代码"
```

**实际执行**（使用 use_subagent）：

```
use_subagent(
    subagent_name="baseline-test",
    task="""
    请审查以下代码：
    
    ```python
    def process_user_input(user_id, amount):
        user = get_user(user_id)
        user.balance += amount
        save_user(user)
        return "Success"
    ```
    
    约束：
    - 不要加载任何 skill
    - 按照你的直觉进行审查
    - 记录你的审查过程
    """,
    background=false
)
```

#### 1.3 记录失败行为

```markdown
**AI 的实际行为**：

1. 快速扫描代码
2. 给出评论："代码看起来还不错，逻辑清晰"
3. 提了一个建议："可以添加注释"

**问题分析**：

| 问题 | 严重性 | AI是否发现 |
|------|--------|-----------|
| 缺少输入验证 | 高 | ❌ 未发现 |
| 缺少错误处理 | 高 | ❌ 未发现 |
| amount 可能为负 | 高 | ❌ 未发现 |
| 并发安全问题 | 中 | ❌ 未发现 |
| 缺少日志记录 | 中 | ❌ 未发现 |
| 返回值不够详细 | 低 | ❌ 未发现 |
| 缺少注释 | 低 | ✅ 发现 |

**AI 的合理化说辞**：
- "代码逻辑清晰，不需要深入检查"
- "这只是简单的函数，应该没问题"
- "用户没有要求检查安全性"
```

### Phase 2: GREEN — 编写 Skill

基于观察到的问题，编写 Skill：

```markdown
---
name: code-review
description: Use when reviewing code for quality, security, and best practices
version: 1.0.0
---

# Code Review Skill

> 系统化的代码审查流程

---

## 🚨 P0 强制规则

> **STOP - 禁止快速扫描后就给结论**

- ❌ **禁止**：看一遍代码就说"看起来不错"
- ❌ **禁止**：只检查功能，忽略其他维度
- ✅ **必须**：按照完整流程逐项检查
- ✅ **必须**：每个维度都给出具体发现

---

## 📋 完整审查流程

### Step 1: 理解代码意图

**必做**：
- [ ] 理解函数/类的目的
- [ ] 识别输入和输出
- [ ] 理解预期的使用场景

### Step 2: 功能正确性检查

- [ ] 逻辑是否正确？
- [ ] 是否处理了边界情况？
  - 空值（null/undefined/None）
  - 空集合（空数组/空字符串）
  - 极值（最大值/最小值）
  - 负值（如果不允许）
- [ ] 是否有潜在的逻辑错误？

### Step 3: 安全性检查

- [ ] 是否验证了输入？
- [ ] 是否有 SQL 注入风险？
- [ ] 是否有 XSS 风险？
- [ ] 是否暴露了敏感信息？
- [ ] 是否有并发安全问题？

### Step 4: 错误处理检查

- [ ] 是否有适当的错误处理？
- [ ] 错误信息是否清晰？
- [ ] 是否捕获了所有可能的异常？
- [ ] 是否有资源泄漏风险？

### Step 5: 性能检查

- [ ] 是否有不必要的计算？
- [ ] 是否有重复查询数据库？
- [ ] 时间复杂度是否合理？
- [ ] 内存使用是否合理？

### Step 6: 可维护性检查

- [ ] 代码是否易读？
- [ ] 命名是否清晰？
- [ ] 是否需要注释？
- [ ] 是否遵循团队规范？

---

### 概述

### 发现的问题

```

### Phase 3: GREEN — 验证通过

运行相同的测试，但这次加载 Skill：

```
use_subagent(
    subagent_name="test-with-skill",
    task="""
    加载 skill: code-review
    
    请审查以下代码：
    
    ```python
    def process_user_input(user_id, amount):
        user = get_user(user_id)
        user.balance += amount
        save_user(user)
        return "Success"
    ```
    """,
    background=false
)
```

**预期结果**：
- ✅ AI 按照 6 个步骤逐项检查
- ✅ 发现了输入验证问题
- ✅ 发现了错误处理问题
- ✅ 发现了 amount 为负的问题
- ✅ 输出格式符合要求

### Phase 4: REFACTOR — 查漏补缺

#### 4.1 设计新的压力场景

**场景2**：提交完美的代码

```python
def calculate_average(numbers: List[float]) -> float:
    """计算平均值，处理空列表和异常值"""
    if not numbers:
        raise ValueError("列表不能为空")
    
    if not all(isinstance(n, (int, float)) for n in numbers):
        raise TypeError("所有元素必须是数字")
    
    return sum(numbers) / len(numbers)
```

**测试**：AI 会不会强行找出"问题"？

**结果**：发现 AI 会说"可以添加异常值检测（如 inf、nan）"

**改进 Skill**：添加"做得好的地方"章节，避免过度挑剔

#### 4.2 设计更刁钻的场景

**场景3**：非常复杂的代码（500行）

**测试**：AI 会不会偷懒只检查前面部分？

**结果**：发现 AI 确实会跳过后半部分

**改进 Skill**：
```markdown
## ⚠️ 特别注意

> **对于长代码（>100行）**：
> 
> - ❌ **禁止**：只检查前面部分就给结论
> - ✅ **必须**：将代码分段，逐段检查
> - ✅ **必须**：最后汇总所有发现
```

#### 4.3 持续迭代

重复 3-5 次：
1. 设计新场景
2. 发现新漏洞
3. 更新 Skill
4. 验证不破坏之前的测试

## 示例2：创建一个"性能优化"Skill

### Phase 1: RED

**基线测试场景**：

```python
# 慢代码示例
def find_duplicates(items):
    duplicates = []
    for i in range(len(items)):
        for j in range(i+1, len(items)):
            if items[i] == items[j] and items[i] not in duplicates:
                duplicates.append(items[i])
    return duplicates
```

**AI 的基线行为**：
- "这个算法看起来不错"
- 没有识别出 O(n²) 时间复杂度
- 没有建议使用 set

### Phase 2: GREEN

编写 Skill：

```markdown
---
name: performance-optimization
description: Use when optimizing code for better performance, identifying bottlenecks, or improving algorithmic complexity
---

# Performance Optimization

## 🔍 性能分析检查清单

### Step 1: 时间复杂度分析
- [ ] 识别所有循环
- [ ] 识别嵌套循环（潜在 O(n²)）
- [ ] 识别递归调用
- [ ] 计算总体时间复杂度

### Step 2: 空间复杂度分析
- [ ] 识别额外的数据结构
- [ ] 识别递归深度
- [ ] 计算总体空间复杂度

### Step 3: 优化机会
- [ ] 能否用哈希表优化查找？
- [ ] 能否用缓存避免重复计算？
- [ ] 能否提前终止循环？
- [ ] 能否并行化？

## 🚨 红旗警告

| 模式 | 问题 | 优化建议 |
|------|------|---------|
| 双重循环 | O(n²) | 使用哈希表 → O(n) |
| `x in list` | O(n) 查找 | 使用 set → O(1) |
| 重复计算 | 浪费 CPU | 使用缓存/记忆化 |
| 逐个append | 多次内存分配 | 预分配空间 |

[...]
```

### Phase 3: GREEN 验证

使用相同的测试代码，验证 AI 现在能：
- ✅ 识别 O(n²) 复杂度
- ✅ 建议使用 set 优化
- ✅ 给出具体的优化代码

### Phase 4: REFACTOR

测试更多场景：
- 已经很优的代码
- 过早优化的陷阱
- 实际性能测试

## 测试技巧

### 1. 设计有代表性的场景

**好的测试场景**：
- ✅ 真实代码，不是玩具示例
- ✅ 包含常见错误模式
- ✅ 有明确的"正确答案"

**不好的测试场景**：
- ❌ 过于简单（Hello World）
- ❌ 过于复杂（无法验证）
- ❌ 没有明确标准

### 2. 记录 AI 的合理化说辞

AI 会用各种理由为错误辩护：

| 说辞 | 现实 |
|------|------|
| "代码看起来不错" | 没有深入检查 |
| "这只是简单的函数" | 简单函数也有问题 |
| "用户没要求检查X" | 应该主动检查 |
| "这是常见写法" | 常见不代表正确 |

**在 Skill 中预见这些说辞，明确禁止**

### 3. 测试边界情况

不只是测试"正常"情况：

- ✅ 空输入
- ✅ 极大输入
- ✅ 完美代码（不要过度挑剔）
- ✅ 糟糕代码（不要放过）

### 4. 迭代测试

不要期望一次成功：

```
第1轮测试 → 发现5个问题 → 更新Skill
第2轮测试 → 发现3个新问题 → 更新Skill
第3轮测试 → 发现1个边缘问题 → 更新Skill
第4轮测试 → 全部通过 → 完成
```

## 测试清单

在发布 Skill 之前，确认：

### 基线测试
- [ ] 运行了基线测试（不加载 Skill）
- [ ] 记录了 AI 的错误行为
- [ ] 识别了根本原因

### Green 测试
- [ ] 加载 Skill 后运行相同测试
- [ ] 验证 AI 不再犯相同错误
- [ ] 验证输出符合预期

### Refactor 测试
- [ ] 至少运行了 3 个不同的压力场景
- [ ] 发现并修复了至少 2 个漏洞
- [ ] 所有之前的测试仍然通过

### 质量检查
- [ ] Description 只描述触发条件（不总结工作流）
- [ ] 有明确的强制规则（STOP/禁止/必须）
- [ ] 有完整的检查清单
- [ ] 有常见错误说明

## 常见测试错误

### 错误1：先写 Skill 再测试

**症状**：Skill 写完了才想起来测试

**问题**：
- 不知道 AI 真正的错误模式
- Skill 可能解决了不存在的问题
- 无法验证 Skill 的有效性

**修复**：
- 删除 Skill
- 先运行基线测试
- 观察失败后再编写

### 错误2：用简单示例测试

**症状**：测试用的都是 `Hello World` 级别的代码

**问题**：
- 无法暴露真实问题
- Skill 在实际使用时会失效

**修复**：
- 使用真实项目的代码
- 包含已知的错误模式
- 测试边界情况

### 错误3：只测试一个场景

**症状**：基线测试通过后就发布

**问题**：
- AI 会找到新的绕过方法
- Skill 有很多漏洞

**修复**：
- 至少测试 3-5 个不同场景
- 包含边界情况
- 持续迭代

### 错误4：不记录失败行为

**症状**：看到 AI 犯错但没记录

**问题**：
- 忘记具体的错误模式
- Skill 无法针对性解决

**修复**：
- 详细记录 AI 的行为
- 记录 AI 的合理化说辞
- 在 Skill 中明确禁止

---

## 总结

**TDD for Skills 的核心原则**：

1. ❌ **没有失败测试 = 没有 Skill**
2. 🔴 **RED**：观察失败是学习的开始
3. 🟢 **GREEN**：让测试通过是最小化解决
4. 🔄 **REFACTOR**：查漏补缺是持续改进

**记住**：
- Skill 是用来防止 AI 犯错的
- 只有看到 AI 犯错，才知道如何防止
- 测试是 Skill 质量的唯一保证

**开始创建你的第一个 Skill**！
