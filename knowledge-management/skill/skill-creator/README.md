## 📋 核心方法：TDD for Skills

```
RED (观察失败)
  ↓
GREEN (编写Skill)
  ↓
REFACTOR (查漏补缺)
```

### 为什么用 TDD？

1. **看到失败才知道教什么**
   - 没有基线测试 = 不知道 AI 真正的错误模式
   - 盲目编写 Skill = 解决不存在的问题

2. **测试是质量的唯一保证**
   - 100% 的高质量 Skill 都经过测试
   - 跳过测试的 Skill 质量差

3. **迭代是发现漏洞的方法**
   - AI 会找到绕过 Skill 的新方法
   - 只有持续测试才能查漏补缺

## 💎 核心设计原则

### 黄金法则

1. **先测试，后编写**（铁律）
2. **Description 只描述触发条件**（不总结工作流）
3. **从简单开始**（1.0 → 2.0 → 3.0 渐进）

### 必备元素

```markdown
✅ 触发场景（何时用 / 何时不用）
✅ P0 强制规则（禁止 / 必须）
✅ 检查清单（- [ ]）
✅ 常见错误（症状 → 修复）
```

## 🧪 测试方法

### 三步测试流程

#### 1. RED - 基线测试

```markdown
目标：观察 AI 在没有 Skill 时的错误行为

方法：
use_subagent(
    task="[测试场景] 约束：不加载任何skill"
)

记录：
- AI 做了什么错误的事
- AI 如何为错误辩护
- 根本原因是什么
```

#### 2. GREEN - 验证测试

```markdown
目标：验证 Skill 能解决观察到的问题

方法：
use_subagent(
    task="[相同场景] 加载skill: [name]"
)

验证：
- AI 不再犯相同错误
- AI 按 Skill 步骤执行
- 输出符合预期
```

#### 3. REFACTOR - 迭代测试

```markdown
目标：发现 AI 绕过 Skill 的新方法

方法：
- 设计更刁钻的场景
- 发现新的合理化说辞
- 更新 Skill 堵住漏洞
- 重复 3-5 次
```

## 🎯 使用场景示例

### 场景1：创建一个新的 Skill

```markdown
用户说："帮我创建一个代码审查的 skill"

执行流程：
1. 加载本 Skill (skill-creator)
2. 回答十问清单
3. 运行基线测试（观察 AI 如何审查代码）
4. 编写 Skill（针对观察到的问题）
5. 运行 green 测试（验证有效性）
6. 迭代 3-5 次（查漏补缺）
```

### 场景2：优化现有 Skill

```markdown
用户说："这个 skill 不太好用，帮我优化一下"

执行流程：
1. 加载本 Skill
2. 分析现有 Skill 的问题
3. 设计压力测试场景
4. 运行测试识别漏洞
5. 更新 Skill
6. 验证不破坏现有功能
```

### 场景3：将工作流转化为 Skill

```markdown
用户说："我们这个流程已经用了3次了，可以做成 skill"

执行流程：
1. 加载本 Skill
2. 提取共同模式
3. 回答十问清单
4. 测试流程的边界情况
5. 编写 Skill
6. 在新场景中验证
```

## 💡 核心洞察

### Skill 的本质

```
Skill = 认知压缩 + 知识结晶 + 执行引擎
```

**类比**：
- Skill 就像压缩包
- 输入：零散的知识、经验、步骤
- 压缩：设计模式、结构化方法
- 输出：可直接执行的工作流

### 知识的三态变化

```
气态：零散知识（难以复用）
  ↓ 凝聚
液态：结构化文档（可查询）
  ↓ 结晶
固态：Skill（可执行）
```
