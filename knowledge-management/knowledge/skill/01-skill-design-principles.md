## 一、Skill 的本质定义

### 1.1 什么是 Skill

**Skill = 可复用的专业工作流 + 领域知识封装**

从示例中总结的核心特征：

| 特征 | 说明 | 示例 |
|------|------|------|
| **专业性** | 封装特定领域的最佳实践 | `product-thinking` 封装产品思维框架 |
| **可复用性** | 跨对话、跨场景适用 | `meta-execution` 适用于所有任务执行 |
| **结构化** | 有明确的流程和步骤 | `knowledge-acquisition` 的五阶段流程 |
| **可组合性** | 可以被其他 skill 调用 | `meta-execution` 被多个技能依赖 |

### 1.2 Skill 的价值

```
传统 AI 对话：
用户问题 → AI 即兴回答 → 质量不稳定

使用 Skill：
用户问题 → 加载 Skill → 遵循专业流程 → 高质量输出
```

**核心价值**：
1. **一致性**：每次执行都遵循相同的标准流程
2. **专业性**：集成领域最佳实践，避免低级错误
3. **可进化**：可以持续优化和版本迭代
4. **可传播**：知识可以跨 Agent、跨用户复用

## 二、Skill 的分层架构

### 2.1 技能类型分类

从示例中归纳出的技能分类：

| 类型 | 定位 | 示例 | 特征 |
|------|------|------|------|
| **元技能（Meta-Skill）** | 通用质量保障框架 | `meta-execution` | 适用于所有任务，自动激活 |
| **领域技能（Domain-Skill）** | 特定领域专业能力 | `product-thinking`, `essence-insight` | 需要特定触发词 |
| **工具技能（Tool-Skill）** | 特定任务的执行流程 | `knowledge-acquisition`, `session-learning` | 明确的输入输出 |
| **方法论技能（Methodology-Skill）** | 思维框架和分析方法 | `karpathy-guidelines`, `brainstorming` | 提供决策指导 |

### 2.2 技能依赖层级

```
┌─────────────────────────────────────────┐
│  Layer 0: 元技能（Meta-Skills）          │
│  - meta-execution (质量保障)             │
│  - 自动激活，贯穿所有任务                 │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  Layer 1: 领域技能（Domain Skills）      │
│  - product-thinking, essence-insight     │
│  - 用户主动触发，提供专业分析             │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  Layer 2: 工具技能（Tool Skills）        │
│  - knowledge-acquisition, self-learning  │
│  - 执行具体任务，产出交付物               │
└─────────────────────────────────────────┘
```

**依赖规则**：
- 下层技能可以依赖上层技能
- 同层技能之间可以组合调用
- 避免循环依赖

## 三、Skill 文件结构设计

### 3.1 标准目录结构

从示例中总结的最佳实践：

```
skill-name/
├── SKILL.md                    # 核心文件（必需）
├── references/                 # 参考资料（可选）
│   ├── framework.md           # 详细框架
│   ├── examples.md            # 示例集合
│   └── checklist.md           # 检查清单
├── templates/                  # 模板文件（可选）
│   ├── output-template.md     # 输出模板
│   └── report-template.md     # 报告模板
├── scripts/                    # 脚本工具（可选）
│   ├── scan.py
│   └── analyze.py
└── patterns/                   # 模式库（可选）
    └── common-pitfalls.md     # 常见陷阱
```

### 3.2 SKILL.md 标准结构

**Frontmatter（YAML 元信息）**：

```yaml
---
name: skill-name                 # 技能标识符（kebab-case）
vibe: "简短的理念描述"            # 可选，一句话概括
description: "详细描述..."        # 触发条件、功能、适用场景
version: 1.0.0                   # 语义化版本号
requires:                        # 依赖的其他技能（可选）
  - meta-execution
parallel_mode: true              # 是否支持并行（可选）
---
```

**正文结构（Markdown）**：

```markdown
# [Skill Name]

> 核心理念/定位的一句话说明

## 技能定位
- 这是什么类型的技能
- 解决什么问题
- 不解决什么问题（边界）

## 触发条件
- 明确的触发词列表
- 适用场景
- 不适用场景（反例）

## 核心原则/理念
- 2-5 条核心原则
- 用金句形式表达

## 执行流程
### Step 1: [步骤名]
- 详细说明
- 输入/输出
- 注意事项

### Step 2: [步骤名]
...

## 关键检查清单
- [ ] 检查项 1
- [ ] 检查项 2

## 常见陷阱/反模式
| 错误做法 | 正确做法 |
|---------|---------|

## 支持文件
- 列出 references/ 等目录的文件

## 版本历史
| 版本 | 日期 | 变更 |
```

## 四、Skill 设计的黄金法则

### 4.1 清晰性原则（Clarity）

**规则 1：一个 Skill 只解决一类问题**

❌ 反例：skill 同时负责"数据分析 + 报告生成 + 邮件发送"
✅ 正例：拆分为三个独立 skill，通过组合使用

**规则 2：触发条件必须明确**

从 `product-thinking` 学到的最佳实践：

```markdown
## 触发条件

✅ 明确的触发词列表：
- 「帮我用产品思维 Review 这个设计」
- 「从用户角度分析一下这个交付物」
- 「这个设计有没有反人性的地方？」

✅ 适用场景：
- 网页、界面、可视化、dashboard
- 文档、报告、分析、总结
- PPT、Keynote、演示文稿

❌ 不适用场景：
- 纯技术问题
- 代码 Debug
- 不涉及用户交付物的内部操作
```

### 4.2 结构化原则（Structure）

**规则 3：流程必须可执行**

从 `meta-execution` 学到的最佳实践：

```markdown
## 执行流程

Step 1: 检测续接信号 → 任一命中 → 进入 Phase 0 流程
Step 2: 识别任务领域 → 从摘要中识别涉及的任务类型/技能
Step 3: 恢复完整上下文 → 加载相关技能文档、读取进度文件
Step 4: 状态声明 → 向用户简述：已完成什么、当前状态、下一步
Step 5: 确认后继续 → 用户确认或给出"继续"指令 → 进入 Phase 1
```

**关键要素**：
- 每步有明确的输入/输出
- 步骤之间有清晰的转换条件
- 支持验证和回退

**规则 4：使用检查清单而非长文本**

❌ 反例：
```markdown
在执行前，你需要理解用户需求，检查是否有相关技能，
回顾历史经验，然后预演方案...
```

✅ 正例（来自 `meta-execution`）：
```markdown
### 🔴 必做核心（每次任务都要过）

□ 需求理解：我真的理解用户想要什么吗？有模糊点吗？
□ 技能检查：有匹配的专业技能可以辅助吗？
□ 经验回顾：从记忆系统中检索与当前任务相关的历史经验
□ 方案预演：脑中走一遍流程，有没有明显的问题？
```

### 4.3 防御性原则（Defense）

**规则 5：显式声明禁止项**

从 `meta-execution` 学到的 **Critical Rules** 模式：

```markdown
## 🚨 Critical Rules（不能绕过的禁止项）

- ❌ **禁止在续接会话时跳过上下文恢复**
- ❌ **禁止跳过启动准备的经验检索**
- ❌ **禁止第2次失败时重复相同操作**
- ❌ **禁止在未过交付审查三阶段前交付**
- ❌ **禁止在遇到阻碍时直接放弃**
```

**规则 6：提供反模式和常见陷阱**

从 `meta-execution` 学到的表格模式：

```markdown
| 不要做 | 应该做 |
|:-------|:-------|
| 看到摘要写"Step 1已完成"就直接从Step 2开始 | 恢复完整流程，确认还有哪些步骤 |
| 用户说"继续"就立刻动手 | 先检测续接信号，恢复上下文 |
| "上个会话做过了，我记得" | 每个新会话都重新加载文档 |
```

### 4.4 可进化原则（Evolution）

**规则 7：版本化和变更记录**

从 `session-learning` 学到的最佳实践：

```markdown
## 版本更新日志

### v3.0.0 (2026-04-29) — 知识提炼深度优化

**核心改进**：
- 🔍 **根因分析 (Step 3.5)**：引入 Five-Why 根因分析
- 📚 **知识层级提升**：从 Layer 2-3 提升到 Layer 4-5
- 🎯 **三层知识架构**：cases → root-causes → principles

**新增目录**：
- `knowledge/root-causes/` — 根因分析结果 (Layer 4)
- `knowledge/principles/` — 通用原则库 (Layer 5)
```

**规则 8：性能和效果度量**

从 `session-learning` 学到的性能对比表：

```markdown
| 对话规模 | 串行模式 | 并行模式 | 加速比 |
|---------|---------|---------|--------|
| **小型** (10-20 轮) | 15s | 8s | 1.9x |
| **中型** (30-50 轮) | 35s | 10s | 3.5x |
| **大型** (60+ 轮) | 70s | 12s | **5.8x** |
```

## 五、高级设计模式

### 5.1 渐进式披露（Progressive Disclosure）

从 `essence-insight` 学到的分层框架：

```markdown
## L1 表象观察
[收集现象，记录关键事件]

## L2 本质规律 + 类比
[归纳规律，找到类比]

## L3→L5 统一三列表
[验证 → 演变 → 预测]
```

**核心思想**：
- 从简单到复杂
- 每层有独立价值
- 支持快速退出

### 5.2 检查点机制（Checkpoint）

从 `meta-execution` 学到的 WAL 协议：

```markdown
### 2-Action Rule
每执行2次查看/搜索后，立即保存关键发现

### WAL Protocol
重要信息先持久化，再继续（内存是易失的）
```

**适用场景**：
- 长任务执行
- 外部信息检索
- 多步骤工作流

### 5.3 并行执行模式

从 `session-learning` 学到的并行优化：

```markdown
## 并行模式（推荐）

触发条件：
- ✅ `use_subagent` 工具可用
- ✅ 对话轮次 ≥ 10
- ✅ `parallel_mode: true`

执行流程：
1. 加载 session 内容到共享上下文
2. 并行启动 7 个专项提取 Agent
3. 等待所有子 Agent 返回结果
4. 合并提取结果
5. 批量写入文件

性能提升：串行 35s → 并行 6s（加速比 5.8x）
```

### 5.4 自适应激活（Adaptive Activation）

从 `meta-execution` 学到的动态激活机制：

```markdown
### 评分标准（C1-C7）

| 编号 | 标准 | 识别信号 |
|:-----|:-----|:---------|
| C1 | 多阶段工作流（3+步骤） | 任务包含多个依赖步骤 |
| C2 | 外部可见交付物 | 产出物会被他人看到 |
| C3 | 涉及部署/发布 | 修改会影响生产环境 |

### 触发规则

命中 ≥ 2 项 → 完整激活（Phase 0-4 全流程）
命中 1 项   → 轻量模式（仅 Phase 3 交付审查）
命中 0 项   → 不激活（简单任务直接执行）
```

## 六、编写 Skill 的实战流程

### 6.1 准备阶段

**Step 1：明确问题域**

问自己5个问题：
1. 这个 skill 要解决什么具体问题？
2. 目标用户是谁？（AI Agent / 人类用户 / 两者）
3. 现有方案的痛点是什么？
4. 成功的标准是什么？
5. 不解决什么问题？（边界）

**Step 2：调研类似 Skill**

- 搜索现有 skill 目录
- 学习相关领域的最佳实践
- 识别可复用的模式

### 6.2 设计阶段

**Step 3：定义核心流程**

使用流程图或伪代码：

```
IF [条件A] THEN
  执行 Step 1
  IF [结果满足条件B] THEN
    执行 Step 2
  ELSE
    回退到 Step 0
ELSE
  跳过，进入 Step 3
```

**Step 4：识别关键决策点**

- 哪里需要用户确认？
- 哪里可能失败？
- 如何处理异常？

**Step 5：设计检查清单**

- 每个步骤的输入/输出验证
- 质量门禁
- 完成度检查

### 6.3 实现阶段

**Step 6：编写 SKILL.md**

按照标准结构：
1. Frontmatter（元信息）
2. 技能定位
3. 触发条件
4. 核心原则
5. 执行流程
6. 检查清单
7. 反模式
8. 支持文件

**Step 7：创建支持文件**

根据需要创建：
- `references/` 详细参考
- `templates/` 输出模板
- `scripts/` 自动化脚本
- `patterns/` 模式库

### 6.4 验证阶段

**Step 8：自检清单**

- [ ] 触发条件是否明确？
- [ ] 流程是否可执行？
- [ ] 是否有反模式说明？
- [ ] 是否有示例？
- [ ] 是否有版本号？
- [ ] 文档是否清晰？

**Step 9：实战测试**

- 在真实场景中使用
- 记录遇到的问题
- 收集用户反馈

**Step 10：迭代优化**

- 更新版本历史
- 补充常见问题
- 优化流程

## 九、常见错误与修正

### 9.1 描述性错误

❌ **错误**：description 太泛化
```yaml
description: "帮助用户完成任务"
```

✅ **正确**：具体说明触发条件和功能
```yaml
description: "产品思维分析技能 — 当用户说「帮我用产品思维看看」「从产品角度 Review 一下」时触发。输出结构化的「场景-用户-欲望-设计」四问分析。"
```

### 9.2 流程性错误

❌ **错误**：流程过于抽象
```markdown
## 执行流程
1. 分析问题
2. 提出方案
3. 验证结果
```

✅ **正确**：具体可执行的步骤
```markdown
## 执行流程

### Step 1: 问题分析
- 使用 `search_memory` 检索相关经验
- 提取关键信息：[具体列表]
- 输出：问题分析报告

### Step 2: 方案设计
- 对照检查清单：[具体项目]
- 生成 2-3 个备选方案
- 输出：方案对比表
```

### 9.3 边界性错误

❌ **错误**：没有明确不适用场景
```markdown
## 适用场景
- 数据分析
- 报告生成
```

✅ **正确**：同时列出适用和不适用
```markdown
## 适用场景
- 结构化数据分析
- 趋势识别和预测

## 不适用场景
- 非结构化文本分析（请使用 text-analysis skill）
- 实时数据流处理（请使用 stream-processing skill）
```

## 十、进阶主题

### 10.1 Skill 的组合与编排

**组合模式**：
```
knowledge-acquisition (主技能)
  └── requires: meta-execution (元技能)
      └── 自动激活，提供质量保障
```

**编排模式**：
```
brainstorming (需求分析)
  ↓
writing-plans (计划编写)
  ↓
executing-plans (执行)
  ↓
verification (验证)
```

### 10.2 Skill 的测试方法

**单元测试**：
- 测试每个步骤的输入输出
- 验证检查清单的完整性
- 测试异常处理

**集成测试**：
- 测试 skill 之间的依赖关系
- 验证组合使用的正确性
- 测试并发场景

**用户测试**：
- 真实场景验证
- 收集用户反馈
- 度量成功率

### 10.3 Skill 的维护策略

**版本管理**：
- 语义化版本号：`major.minor.patch`
- 重大变更：major +1
- 新功能：minor +1
- Bug 修复：patch +1

**废弃策略**：
```markdown
---
name: old-skill
deprecated: true
deprecated_message: "已被 new-skill 替代，请使用 new-skill"
replacement: new-skill
---
```

**兼容性保障**：
