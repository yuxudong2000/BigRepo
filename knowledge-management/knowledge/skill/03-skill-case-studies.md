## 案例1: meta-execution

> **一次做对的元技能 — 通用任务执行质量保障框架**

### 基本信息

```yaml
---
name: meta-execution
type: 元技能 (Meta-Skill)
version: 1.0.0
complexity: ⭐⭐⭐⭐
lines_of_code: ~400 行
support_files: 3 个参考文档
---
```

### 设计目标

**核心理念**：把事情一次做对，而不是反复返工

**解决的问题**：
1. 任务执行缺乏标准流程
2. 常见错误重复发生
3. 质量依赖临时发挥
4. 知识无法沉淀复用

### 架构剖析

#### 1. 五阶段生命周期

```
Phase 0: 续接恢复 (Resume)
   ↓
Phase 1: 启动准备 (Prepare)
   ↓
Phase 2: 执行过程 (Execute)
   ↓
Phase 3: 交付审查 (Review)
   ↓
Phase 4: 事后复盘 (Reflect)
```

**设计亮点**：
- ✅ 覆盖任务完整生命周期
- ✅ 每个阶段有明确目标和检查项
- ✅ 支持跳过非必要阶段（轻量模式）

#### 2. 自适应激活机制

```markdown
评分标准（C1-C7）：
- C1: 多阶段工作流（3+步骤）
- C2: 外部可见交付物
- C3: 涉及部署/发布
- C4: 无人值守执行
- C5: 数据敏感性要求
- C6: 用户强调质量
- C7: 高复杂度任务

触发规则：
命中 ≥ 2 项 → 完整激活
命中 1 项   → 轻量模式
命中 0 项   → 不激活
```

**设计亮点**：
- ✅ 避免过度设计（简单任务不触发）
- ✅ 按需激活（复杂任务全流程）
- ✅ 灵活性高（可扩展更多标准）

#### 3. Critical Rules 机制

```markdown
## 🚨 Critical Rules（不能绕过的禁止项）

- ❌ **禁止在续接会话时跳过上下文恢复**
- ❌ **禁止跳过启动准备的经验检索**
- ❌ **禁止第2次失败时重复相同操作**
- ❌ **禁止在未过交付审查三阶段前交付**
- ❌ **禁止在遇到阻碍时直接放弃**
```

**设计亮点**：
- ✅ 显式声明禁止项（不是建议，是规则）
- ✅ 使用强烈的视觉标记（🚨, ❌）
- ✅ 说明后果（"将导致XX问题"）

#### 4. 三阶段交付审查

```
Stage 1: 规格符合
  ├─ 完整性：用户要求都实现了吗？
  ├─ 无遗漏：附加要求都完成了吗？
  └─ 无越界：没做用户没要求的事？
       ↓
Stage 2: 质量审查
  ├─ 正确性：逻辑正确吗？
  ├─ 清晰性：用户能看懂吗？
  └─ 美观性：格式整齐吗？
       ↓
Stage 3: 输出规范
  ├─ 网页/HTML：需要预览吗？
  ├─ 图片：需要展示吗？
  └─ 代码：能直接运行吗？
```

**设计亮点**：
- ✅ 层层把关（前一阶段不通过，后续不执行）
- ✅ 覆盖全面（规格、质量、输出三维度）
- ✅ 可操作性强（每项都有明确检查标准）

### 关键代码片段

**启动准备检查清单**：

```markdown
### 🔴 必做核心（每次任务都要过）

□ 需求理解：我真的理解用户想要什么吗？有模糊点吗？
□ 技能检查：有匹配的专业技能可以辅助吗？
□ 经验回顾：从记忆系统中检索与当前任务相关的历史经验
□ 方案预演：脑中走一遍流程，有没有明显的问题？

### 🟡 复杂任务扩展（按需启用）

□ 苏格拉底式提问：先问清范围/预期/约束，再动手
□ 设计先行：分块展示方案，确认后再执行
□ 任务原子化：拆分为可独立验证的小任务
```

**3-Strike 错误处理**：

```markdown
第1次失败: 诊断 & 修复 → 识别根因 → 定向修复
第2次失败: 换方法 → 不同工具/路径/方案（⛔ 不重复相同失败操作）
第3次失败: 资源组合 → 质疑假设 + 搜索方案 + 组合方法
第4-9次: Relentless Mode → 尝试5-10种不同方案
第10次后: 升级给用户 → 列出尝试过的方案 + 具体错误
```

### 学到的经验

#### ✅ 做得好的地方

1. **分阶段设计清晰**
   - 每个阶段有独立价值
   - 可以单独使用（轻量模式）
   - 易于理解和记忆

2. **强制性与灵活性并存**
   - Critical Rules 确保底线
   - 自适应激活避免过度
   - 支持手动覆盖

3. **防御性编程思想**
   - 预见常见错误
   - 提供反模式说明
   - 强制验证关键步骤

4. **可操作性强**
   - 每个检查项都明确
   - 流程可直接执行
   - 有具体的通过标准

#### ⚠️ 可能的局限

1. **文档长度**：~400行，初学者可能觉得复杂
   - **改进建议**：提供"快速开始"版本

2. **上下文消耗**：完整激活可能占用较多 token
   - **改进建议**：支持按需加载章节

3. **学习曲线**：需要理解完整框架
   - **改进建议**：提供分级学习路径

### 适用场景

✅ **适合**：
- 复杂的多步骤任务
- 质量要求高的交付物
- 需要标准化流程的场景
- 作为其他 Skill 的基础框架

❌ **不适合**：
- 极简单的单步操作
- 探索性的快速尝试
- 时间敏感的紧急任务

### 可复用的设计模式

1. **五阶段生命周期模式** → 可用于任何任务执行
2. **自适应激活模式** → 可用于智能触发
3. **Critical Rules 模式** → 可用于质量保障
4. **三阶段审查模式** → 可用于交付验证
5. **3-Strike 错误处理** → 可用于容错设计

## 案例2: session-learning

> **对话知识沉淀技能（v3.0 深度优化版）**

### 基本信息

```yaml
---
name: session-learning
type: 工具技能 (Tool-Skill)
version: 3.0.0
complexity: ⭐⭐⭐⭐⭐
lines_of_code: ~800+ 行
support_files: 多个参考文档和脚本
parallel_mode: true
deep_analysis: true
---
```

### 设计目标

**核心理念**：将对话完整归档并自动提炼可复用知识，形成持久化个人知识资产

**解决的问题**：
1. 对话历史无法有效沉淀
2. 经验无法跨会话复用
3. 知识提炼耗时长
4. 深度洞察难以提取

### 架构剖析

#### 1. 三层知识架构（v3.0）

```
Layer 1: Sessions (原始对话完整存档)
   ↓
Layer 2-3: Solutions/Tools/Patterns/Insights (具体解法和经验)
   ↓
Layer 4: Root-Causes (根因分析)
   ↓
Layer 5: Principles (通用原则库)
```

**设计亮点**：
- ✅ 多层级存储（从具体到抽象）
- ✅ 层层提炼（知识价值递增）
- ✅ 跨 session 可复用

#### 2. 并行执行优化（v2.0 核心创新）

**串行模式（v1.0）**：
```
for category in [solutions, tools, patterns, ...]:
    extract(category)  # 串行执行，总耗时 35s
```

**并行模式（v2.0）**：
```
tasks = []
for category in [solutions, tools, patterns, ...]:
    task = use_subagent(category)  # 并发启动
    tasks.append(task)
await gather(tasks)  # 等待全部完成，总耗时 6s
```

**性能提升**：
```
| 对话规模 | 串行模式 | 并行模式 | 加速比 |
|---------|---------|---------|--------|
| 小型    | 15s     | 8s      | 1.9x   |
| 中型    | 35s     | 10s     | 3.5x   |
| 大型    | 70s     | 12s     | 5.8x ⚡ |
```

**设计亮点**：
- ✅ 显著性能提升（5.8x）
- ✅ 自动降级兜底（工具不可用时回退串行）
- ✅ 流式进度反馈（实时展示进度）

#### 3. 根因分析引擎（v3.0 深度创新）

**Five-Why 分析流程**：

```markdown
Step 1: 筛选值得分析的 insights
  ├─ ✅ 涉及问题诊断/错误修复
  ├─ ✅ 涉及设计决策/权衡取舍
  ├─ ✅ 涉及意外发现/认知更新
  └─ ❌ 纯信息记录类（跳过）

Step 2: 对每个 insight 执行 Five-Why
  use_subagent(
      subagent_name="root-cause-analyzer",
      task="执行根因分析..."
  )

Step 3: 写入根因分析结果
  → knowledge/root-causes/YYYY-MM-DD_<topic>.md

Step 4: 跨 session 原则提炼（每月执行）
  ├─ 扫描 root-causes/ 目录
  ├─ 识别重复出现的原则（≥3次，不同 session）
  └─ 写入 knowledge/principles/<principle>.md
```

**输出示例**：

```markdown
# 根因分析: Skill 优化路径混淆问题

## Five-Why 分析

### Why 1: 为什么在错误路径验证?
→ 验证逻辑使用相对路径,依赖当前 cwd

### Why 2: 为什么依赖 cwd?
→ 操作未显式指定目标上下文

### Why 3: 为什么未显式指定?
→ 用户指令"优化 skill"存在歧义

### Why 4: 为什么存在歧义?
→ 指令隐式依赖上下文假设

### Why 5: 为什么会有隐式假设?
→ 缺少"多上下文操作前必须消歧"的设计原则

## 根本原因
**隐式上下文依赖** — 操作结果依赖未显式声明的状态

## 通用原则
### 显式依赖原则 (Explicit Dependency Principle)
> 操作结果应只依赖显式参数,不依赖隐式状态
```

**设计亮点**：
- ✅ 从症状追溯到根本原因
- ✅ 提炼跨领域可复用原则
- ✅ 建立原则库（长期价值）

#### 4. 原子写入保护

**问题**：并发写入多个文件，如何保证数据安全？

**解决方案**：临时文件 + 原子替换

```python
async def atomic_write_one(path: Path, content: str):
    async with semaphore:
        # 1. 临时文件
        temp = path.parent / f".tmp_{path.name}_{uuid.uuid4().hex[:8]}"
        # 2. 写入 + fsync（确保数据落盘）
        await asyncio.to_thread(temp.write_text, content)
        # 3. 原子替换（os.replace 是原子操作）
        await asyncio.to_thread(os.replace, temp, path)
```

**设计亮点**：
- ✅ 防止写入中断导致文件损坏
- ✅ 支持并发写入（信号量限流）
- ✅ 自动清理临时文件

### 关键代码片段

**并行提取触发条件**：

```markdown
## 并行模式（优先）

触发条件：
- ✅ `use_subagent` 工具可用
- ✅ 对话轮次 ≥ 10（小对话自动降级）
- ✅ `parallel_mode: true`（Skill frontmatter 配置）

自动降级条件（满足任一即降级）：
1. `use_subagent` 工具不可用
2. 对话轮次 < 10
3. 任意子 Agent 超时（> 30s）
4. `parallel_mode: false`（手动禁用）
```

**流式进度反馈**：

```markdown
🚀 开始知识提炼（7 个分类并行）...

✅ [solutions] 提炼完成 → 1 个文件
✅ [tools] 提炼完成 → 1 个文件
✅ [patterns] 无可提炼内容
✅ [insights] 提炼完成 → 1 个文件
✅ [memories] 提炼完成 → 2 条候选
✅ [rules] 无可提炼内容
✅ [skills] 提炼完成 → 1 个文件

📝 批量写入文件中...（4 个文件并发）
✅ 全部写入完成（耗时 1.2s）

⚡ **性能统计**
  - 总耗时：8.5s（串行模式预计 40s，节省 79%）
  - 知识提炼：6.2s（并行）
  - 文件写入：1.2s（批量）
```

### 学到的经验

#### ✅ 做得好的地方

1. **版本迭代清晰**
   - v1.0: 基础功能
   - v2.0: 并行优化（5.8x 加速）
   - v3.0: 深度分析（根因+原则）
   - 每个版本都有明确的价值增量

2. **性能优化显著**
   - 识别性能瓶颈（串行提取）
   - 引入并行机制（use_subagent）
   - 提供降级兜底（自动回退）
   - 展示性能数据（用户可感知）

3. **知识层级提升**
   - Layer 1-3: 具体经验（解法/工具/模式）
   - Layer 4: 根因分析（为什么）
   - Layer 5: 通用原则（跨领域复用）
   - 知识价值逐层递增

4. **工程质量高**
   - 原子写入保护（数据安全）
   - 并发控制（信号量限流）
   - 错误恢复（临时文件清理）
   - 完整的错误处理

#### ⚠️ 可能的局限

1. **复杂度高**：800+ 行代码，理解成本高
   - **改进建议**：提供简化版（只保留核心功能）

2. **依赖子 Agent**：并行模式需要 `use_subagent` 工具
   - **改进建议**：文档中明确说明依赖

3. **根因分析耗时**：增加 15s 执行时间
   - **改进建议**：提供开关（按需启用）

### 适用场景

✅ **适合**：
- 长对话的知识沉淀
- 需要高性能处理的场景
- 深度学习和复盘
- 建立个人知识库

❌ **不适合**：
- 简短对话（轮次<10）
- 纯信息查询对话
- 无沉淀价值的对话

### 可复用的设计模式

1. **并行执行模式** → 适用于所有独立任务的批量处理
2. **降级兜底模式** → 适用于需要容错的系统
3. **原子写入模式** → 适用于并发文件操作
4. **根因分析模式** → 适用于问题诊断和深度学习
5. **版本演进模式** → 如何渐进式增强功能

## 案例3: essence-insight

> **本质洞察力 — 透过现象看本质的元能力**

### 基本信息

```yaml
---
name: essence-insight
type: 领域技能 (Domain-Skill)
version: 2.5.0
complexity: ⭐⭐⭐⭐
lines_of_code: ~636 行
support_files: 2 个参考文档
requires:
  - meta-execution
---
```

### 设计目标

**核心理念**：透过现象看本质，抽丝剥茧找到底层规律

**解决的问题**：
1. 分析流于表面，难以触及本质
2. 洞察难以表达和传播
3. 预测缺乏逻辑支撑
4. 知识积累无法转化为洞察力

### 架构剖析

#### 1. 五层洞察闭环框架

```
L1: 表象观察（观察）
  ↓ 归纳
L2: 本质规律 + 类比（假说）
  ↓ 验证
L3: 规律映射 ⭐（验证）
  ↓ 演变分析
L4: 规律演变（预测第一层）
  ↓ 现实推演
L5: 现实预测（预测第二层）
```

**设计亮点**：
- ✅ 完整的科学方法论（观察→假说→验证→预测）
- ✅ L3 是关键升级（用规律逐一解释表象）
- ✅ 双层预测（规律本身如何变 + 现实如何变）

#### 2. 统一三列表（v2.2 核心创新）

**维度一致性原则**：
> 三列表的左列不是"描述每行在说什么"，而是 L2 进化序列的阶段标签

**错误做法（v2.1）**：
```
| L3左列 | L4左列 | L5左列 |
|--------|--------|--------|
| 协作化 | 意图竞争 | 新维度 |  ← 每段各讲各的，互不相通
```

**正确做法（v2.2）**：
```
| 阶段 | 现实 | 类比 |
|------|------|------|
| ✅ L3 已验证 |
| 1.0 工具 | Copilot Tab补全 | 学徒填色 |
| 2.0 自主 | Claude自主完成 | 熟练工干活 |
| 3.0 工业化 | Codex多Agent | 流水线工厂 |
| ──── 虚线分隔 ──── |
| 🔄 L4 各阶段演变 |
| 1.0 工具 | 补全型消亡 | 学徒被替代 |
| 2.0 自主 | 单Agent成基础 | 熟练工成标配 |
| 3.0 工业化 | 监督→自动化 | 流水线→无人工厂 |
| ──── 虚线分隔 ──── |
| 🔮 L5 新阶段 |
| 4.0 全自动 | 24/7无人值守 | 无人工厂 |
| 5.0 自进化 | AI自主优化 | 工厂自己升级 |
```

**设计亮点**：
- ✅ 左列贯穿L2阶段标签（一条进化线）
- ✅ 类比贯穿全表（每行都有）
- ✅ 三段递进（验证→演变→预测）

#### 3. 类比驱动的表达

**核心原则**：
> 如果你真的看到了本质，那它一定能被类比到熟悉的场景

**好类比的标准**：

| 标准 | 说明 | 示例 |
|------|------|------|
| 受众熟悉 | 目标受众一定知道 | ✅ 翻译官 ❌ 量子纠缠 |
| 结构相似 | 内在结构/关系相似 | AI产品 ≈ 翻译官 |
| 简洁生动 | 一句话能说清楚 | "AI就像压缩包" |
| 有启发性 | 帮助理解本质 | 压缩→理解→泛化 |

**类比构建方法**：
```
1. 提炼核心结构 → AI产品在"翻译"用户意图
2. 搜索熟悉场景 → 翻译官、同声传译
3. 验证结构匹配 → 用户→AI→代码 ≈ 说话→翻译官→外语
4. 简化表达 → "AI编程工具就像翻译官"
5. 检验理解 → 听众会"哦～原来如此"吗？
```

#### 4. 分发vs本质检验法（v2.3.2）

**检验问题**：你的洞察回答的是哪个问题？

| 问题类型 | 示例 | 判断 |
|---------|------|------|
| 在哪/怎么分发 | "AI从独立App变成嵌入式功能" | ❌ 只是分发形态 |
| 做了什么不同的事 | "AI产品卖的是认知代劳" | ✅ 触及本质 |

**检验方法**：用"即使……也……"句式测试

- ❌ "即使AI从独立App变成嵌入式，它做的事情本质上没变"
- ✅ "即使AI换一种分发形式，它依然是在替用户思考"

### 关键代码片段

**对外写作呈现规范（v2.3）**：

```markdown
## 对外写作（读者友好）

❌ 内部标签直接暴露（技术感）：
  L1 表象观察
  L2 本质规律
  L3 规律映射
  L4 规律演变
  L5 现实预测

✅ 对外写作（自然阅读）：
  ① 表象观察
  ② 本质规律 — 核心洞察
  ③ 趋势推演（含验证 / 演变 / 预测三段）
```

**维度一致性检查**：

```markdown
输出三列表前，对照检查：
- [ ] L3的左列是否直接使用L2的进化阶段标签？
- [ ] L4的左列是否和L3**完全一致**？
- [ ] L5的左列是否从L2最高阶段**自然延伸**？
- [ ] 从上到下读左列，是否能看到一条**连贯的进化线**？
```

### 学到的经验

#### ✅ 做得好的地方

1. **方法论完整**
   - 五层框架覆盖完整分析流程
   - 对应科学方法（观察→假说→验证→预测）
   - 每层有独立价值，可单独使用

2. **类比贯穿始终**
   - 不只在 L2 出现一次
   - 每个阶段都有对应类比
   - 类比让抽象概念可理解

3. **版本迭代清晰**
   - v2.1: 发现问题（左列随意贴标签）
   - v2.2: 解决问题（维度一致性原则）
   - v2.3: 呈现优化（对外写作规范）

4. **细节打磨到位**
   - 提供检验法（分发vs本质）
   - 提供检查清单（12项）
   - 提供常见陷阱说明

#### ⚠️ 可能的局限

1. **学习曲线陡峭**：五层框架需要深入理解
   - **改进建议**：提供简化版（三层框架）

2. **执行成本高**：完整分析需要大量思考
   - **改进建议**：按需使用（不是每个分析都需要L5）

3. **对知识储备要求高**：没有足够知识无法洞察
   - **改进建议**：明确知识积累的阈值

### 适用场景

✅ **适合**：
- 复杂现象的深度分析
- 趋势预测和战略决策
- 知识提炼和原则抽象
- 需要说服他人的场景（类比帮助理解）

❌ **不适合**：
- 知识积累不足的领域
- 简单问题的快速分析
- 时间紧迫的场景

### 可复用的设计模式

1. **五层洞察框架** → 适用于任何复杂分析
2. **统一三列表** → 适用于多维度对比展示
3. **类比驱动表达** → 适用于抽象概念的传播
4. **维度一致性原则** → 适用于保持逻辑连贯
5. **对外呈现优化** → 适用于面向读者的交付物

## 案例4: product-thinking

> **产品思维分析技能 — 用产品经理视角审视交付物**

### 基本信息

```yaml
---
name: product-thinking
type: 领域技能 (Domain-Skill)
version: 3.0.0
complexity: ⭐⭐⭐
lines_of_code: ~300 行
support_files: 3 个参考文档
---
```

### 设计目标

**核心理念**：每个交付物都是给别人用的，都可以理解成一种"产品"

**解决的问题**：
1. 从实现者视角出发，而非用户视角
2. 功能堆砌，缺少场景思考
3. 交付物难以被用户接受
4. 缺少系统的产品分析方法

### 架构剖析

#### 1. 四问框架（逆向分析）

```
问题4：场景（最外层）
  → 什么时候、在哪里使用？
  → 用什么方式？花多长时间？
       ↓
问题3：用户画像（第二层）
  → 用户带着什么预期来的？
  → 他想完成什么任务？
       ↓
问题2：欲望/痛点（第三层）
  → 满足感：想获得什么"爽"？
  → 恐惧感：想消除什么焦虑？
       ↓
问题1：设计决策（核心）
  → 基于以上，优先呈现什么？
  → 主动省略/弱化什么？
  → 用户获取价值的路径如何最短？
```

**设计亮点**：
- ✅ 从外到内递进（场景→用户→痛点→设计）
- ✅ 每层分析支撑下一层
- ✅ 最终输出可执行的设计建议

#### 2. 视角转换机制

**核心转换**：

| 实现者视角 ❌ | 用户服务视角 ✅ |
|-------------|----------------|
| 我做一个能力树可视化 | 用户想快速了解AI助手的能力变化 |
| 我做一个神经网络布局 | 用户想一眼看到关键的新增/变化 |
| 我把三棵树都展示出来 | 用户在30秒内想看到"今天有什么不同" |

**语言转换**：
```
❌ "我能提供什么产品"
✅ "用户要什么服务 + 我能如何提供"
```

#### 3. 反模式诊断

**症状-诊断-修复模式**：

```markdown
### ❌ 功能堆砌

**症状**：把所有能做的功能都展示出来
**诊断**：没有问"用户80%的场景需要哪20%的功能"
**修复**：识别核心场景，其他功能隐藏或弱化

### ❌ 视觉炫技

**症状**：优先考虑"这个效果很酷"
**诊断**：从实现者视角出发，而非用户视角
**修复**：问"这个效果帮助用户完成什么任务？"

### ❌ 信息过载

**症状**：一个页面塞太多内容
**诊断**：没有区分信息优先级
**修复**：根据场景确定默认展示什么、折叠什么、隐藏什么
```

### 关键代码片段

**输出格式模板**：

```markdown
## 🔍 产品思维分析

### 场景判断
[描述用户使用场景、时间、设备、频率]

### 用户画像
[描述目标用户的预期、任务、背景]

### 欲望/痛点
- 满足感：[用户想获得什么]
- 恐惧感：[用户想消除什么]

### 设计决策建议
- ✅ 应该突出：[...]
- ❌ 应该弱化/删除：[...]
- 🔄 交互路径优化：[...]

### 发现的反模式
- [具体问题 + 修复建议]
```

**可操作改进清单**：

```markdown
1. **如果用户只有30秒，他能看到最重要的信息吗？**
2. **用户完成核心任务需要几次点击？能减少吗？**
3. **页面上有没有"我觉得酷但用户用不到"的东西？**
4. **如果让用户选择，他会保留哪些、删掉哪些？**
5. **这个设计解决了用户的什么痛点或满足了什么爽点？**
```

### 学到的经验

#### ✅ 做得好的地方

1. **框架简单有效**
   - 四问框架易于理解
   - 层次递进逻辑清晰
   - 可立即应用

2. **视角转换到位**
   - 明确区分销售思维vs产品思维
   - 提供具体的转换示例
   - 用金句强化记忆

3. **反模式丰富**
   - 覆盖常见错误
   - 提供具体修复方案
   - 易于自我检查

4. **输出结构清晰**
   - 模板化输出格式
   - 可操作的建议清单
   - 便于后续执行

#### ⚠️ 可能的局限

1. **适用范围**：主要针对B端工具和内部交付物
   - **改进建议**：补充C端产品的特殊考虑

2. **深度权衡**：为了简洁，省略了很多细节
   - **改进建议**：在references/中提供完整版

3. **执行依赖**：需要有产品思维基础
   - **改进建议**：提供基础概念入门

### 适用场景

✅ **适合**：
- 网页、界面、可视化设计
- 文档、报告、分析评审
- PPT、演示文稿优化
- 任何需要交付给他人的产物

❌ **不适合**：
- 纯技术问题（算法优化等）
- 代码 Debug
- 不涉及用户交付物的内部操作

### 可复用的设计模式

1. **四问框架** → 适用于任何用户导向的分析
2. **视角转换表** → 适用于纠正思维模式
3. **反模式诊断** → 适用于质量检查
4. **金句强化** → 适用于核心原则传播

## 案例5: brainstorming

> **需求分析与设计验证技能**

### 基本信息

```yaml
---
name: brainstorming
type: 方法论技能 (Methodology-Skill)
version: [未明确标注]
complexity: ⭐⭐⭐
lines_of_code: ~165 行
support_files: 1 个（visual-companion.md）
---
```

### 设计目标

**核心理念**：在任何创造性工作前，先理解需求、探索设计、获得批准

**解决的问题**：
1. 直接开始实现，没有设计阶段
2. 需求理解偏差导致返工
3. 多个方案无法系统对比
4. 设计缺少用户确认

### 架构剖析

#### 1. 硬门禁设计

```markdown
<HARD-GATE>
Do NOT invoke any implementation skill, 
write any code, scaffold any project, 
or take any implementation action 
until you have presented a design 
and the user has approved it.

This applies to EVERY project 
regardless of perceived simplicity.
</HARD-GATE>
```

**设计亮点**：
- ✅ 使用强制性标签（<HARD-GATE>）
- ✅ 适用所有项目（无例外）
- ✅ 明确禁止行为列表

#### 2. 九步检查清单

```markdown
1. **Explore project context** — 检查文件、文档、提交历史
2. **Offer visual companion** — 如需要，提供可视化工具
3. **Ask clarifying questions** — 一次一个，理解需求
4. **Propose 2-3 approaches** — 提供选项和权衡
5. **Present design** — 分段展示，每段确认
6. **Write design doc** — 保存到指定位置
7. **Spec self-review** — 自检占位符、矛盾、模糊
8. **User reviews written spec** — 用户审查设计文档
9. **Transition to implementation** — 调用 writing-plans
```

**设计亮点**：
- ✅ 每步有明确动作
- ✅ 包含验证点（步骤8）
- ✅ 明确下一步（步骤9）

#### 3. 反模式预防

```markdown
## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. 
A todo list, a single-function utility, a config change — all of them. 

"Simple" projects are where unexamined assumptions 
cause the most wasted work. 

The design can be short (a few sentences for truly simple projects), 
but you MUST present it and get approval.
```

**设计亮点**：
- ✅ 直接点名常见心态
- ✅ 解释为什么简单项目也需要设计
- ✅ 提供灵活性（设计可以很短）

#### 4. 流程图可视化

使用 Graphviz dot 语法描述流程：

```dot
digraph brainstorming {
    "Explore project context" -> "Visual questions ahead?";
    "Visual questions ahead?" -> "Offer Visual Companion" [label="yes"];
    "Visual questions ahead?" -> "Ask clarifying questions" [label="no"];
    ...
}
```

### 学到的经验

#### ✅ 做得好的地方

1. **门禁严格**
   - HARD-GATE 确保流程不被跳过
   - 适用所有项目（无例外）
   - 明确禁止行为

2. **流程完整**
   - 覆盖从需求到实现的完整路径
   - 每步有验证点
   - 终止条件明确

3. **防御性设计**
   - 预见常见心态（"太简单不需要设计"）
   - 提供反驳理由
   - 保持灵活性（简单设计也可以很短）

4. **可视化流程**
   - 使用流程图展示逻辑
   - 易于理解和记忆
   - 便于检查遗漏

#### ⚠️ 可能的局限

1. **流程较重**：对于非常简单的任务可能过度
   - **改进建议**：提供轻量级快速通道

2. **依赖用户配合**：需要用户审查设计
   - **改进建议**：说明如何引导用户

### 适用场景

✅ **适合**：
- 创造性工作（新功能、新组件）
- 需求不明确的任务
- 涉及用户交互的设计
- 可能有多种实现方案的场景

❌ **不适合**：
- 紧急 Bug 修复
- 明确的重复性任务
- 纯探索性实验

### 可复用的设计模式

1. **硬门禁模式** → 适用于强制流程控制
2. **九步检查清单** → 适用于需求到实现流程
3. **反模式预防** → 适用于预见常见错误
4. **流程图可视化** → 适用于复杂流程说明

## 案例6: karpathy-guidelines

> **行为准则 — 减少常见LLM编码错误**

### 基本信息

```yaml
---
name: karpathy-guidelines
type: 方法论技能 (Methodology-Skill)
version: [未标注]
complexity: ⭐⭐
lines_of_code: ~68 行
license: MIT
---
```

### 设计目标

**核心理念**：简洁、精确、克制

**解决的问题**：
1. AI代码过度复杂
2. 改动范围过大
3. 假设未显式表达
4. 缺少可验证的成功标准

### 架构剖析

#### 1. 四大原则

```markdown
1. Think Before Coding
   - 显式表达假设
   - 多种解释时，提出所有选项
   - 简单方案优先
   - 不确定时，停下来问

2. Simplicity First
   - 最少代码解决问题
   - 无推测性功能
   - 无不必要的抽象
   - 200行能写成50行，就重写

3. Surgical Changes
   - 只改必须改的
   - 不"改进"临近代码
   - 遵循现有风格
   - 每个改动都能追溯到用户需求

4. Goal-Driven Execution
   - 定义可验证的成功标准
   - 分步执行，每步验证
   - 强成功标准让你能独立循环
```

**设计亮点**：
- ✅ 每个原则一句话总结
- ✅ 具体可执行的子规则
- ✅ 反例和正例对比

#### 2. 极简风格

整个 Skill 只有 68 行，但包含：
- 4 个核心原则
- 每个原则的详细说明
- 可操作的检查标准
- 权衡说明

**设计亮点**：
- ✅ 高度浓缩，无废话
- ✅ 易于快速扫描
- ✅ 便于记忆和应用

#### 3. 反思性检查

```markdown
Ask yourself: 
"Would a senior engineer say this is overcomplicated?" 

If yes, simplify.
```

**设计亮点**：
- ✅ 引导自我检查
- ✅ 用"资深工程师"作为标准
- ✅ 提供明确的行动（simplify）

### 关键代码片段

**Surgical Changes 的检查标准**：

```markdown
The test: 
Every changed line should trace directly 
to the user's request.
```

**Goal-Driven Execution 的转换示例**：

```markdown
Transform tasks into verifiable goals:
- "Add validation" 
  → "Write tests for invalid inputs, then make them pass"
  
- "Fix the bug" 
  → "Write a test that reproduces it, then make it pass"
  
- "Refactor X" 
  → "Ensure tests pass before and after"
```

### 学到的经验

#### ✅ 做得好的地方

1. **极致简洁**
   - 68行包含完整原则
   - 无多余解释
   - 直击要点

2. **可操作性强**
   - 每个原则都有具体检查标准
   - 提供转换示例
   - 易于自我验证

3. **来源明确**
   - 基于 Andrej Karpathy 的观察
   - 提供原始链接
   - 说明权衡（偏向谨慎）

4. **设计哲学清晰**
   - 简洁优先
   - 显式优于隐式
   - 克制优于过度

#### ⚠️ 可能的局限

1. **适用范围**：主要针对代码编写
   - **改进建议**：明确非代码场景的适用性

2. **文化差异**：基于"资深工程师"标准
   - **改进建议**：提供更多判断依据

### 适用场景

✅ **适合**：
- 代码编写
- 代码审查
- 重构优化
- 任何需要克制和简洁的场景

❌ **不适合**：
- 探索性原型开发
- 需要快速迭代的场景
- 创意性项目（初期）

### 可复用的设计模式

1. **极简设计** → 适用于需要快速理解的文档
2. **反思性检查** → 适用于自我验证
3. **转换示例** → 适用于概念到行动的桥接
4. **权衡说明** → 适用于建立正确预期

## 提炼的通用经验

### 1. 版本演进策略

从 `session-learning` 学到的版本演进：
```
v1.0 → 基础功能（串行执行）
v2.0 → 性能优化（并行执行，5.8x 加速）
v3.0 → 深度提升（根因分析 + 原则提炼）
```

**经验**：
- ✅ 每个版本有明确价值增量
- ✅ 保持向后兼容（降级兜底）
- ✅ 记录性能数据（可感知的提升）

### 2. 文档长度权衡

| 文档长度 | 优势 | 劣势 | 适用场景 |
|---------|------|------|---------|
| 极简（<100行） | 易于理解 | 缺少细节 | 原则型、检查清单型 |
| 适中（200-400行） | 平衡 | 需要一定时间 | 领域技能、方法论 |
| 完整（500+行） | 全面 | 学习成本高 | 复杂工具、元技能 |

**经验**：
- ✅ 优先简洁，按需扩展
- ✅ 使用渐进式披露（快速参考 + 详细说明）
- ✅ 提供多个入口（快速开始 + 完整指南）
